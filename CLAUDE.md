## Key Decisions (read before writing code)

- **No MediatR.** Use `ICommandHandler<T>` / `IQueryHandler<T, TResult>` from `MyMarina.Application.Abstractions`. Cross-cutting concerns via Scrutor decorators.
- **Multi-context user system:** A User is a human who can have multiple roles across different tenants and marinas via `UserContext` junction records. At login, the user selects which context to use; JWT is scoped to that context. `UserContext.MarinaId` nullability determines permission scope: null = tenant-level (sees all marinas in tenant), set = marina-level (sees only that marina). Platform operators can switch between any context; tenant owners/staff can switch between contexts within their roles; customers see unified view across marinas (no context switching). Use `has_multiple_contexts` JWT claim to show context-switch UI.
- **Multi-tenancy:** Shared DB, shared schema, EF Core global query filters on `TenantId`. Three-level scoping: `ITenantContext` (cross-tenant) + `IMarinaContext` (per-marina staff) + `ICustomerContext` (customer portal). All three implemented by `HttpTenantContext` in Infrastructure — resolved from JWT claims. Register each interface separately via Scrutor pointing to the same `HttpTenantContext` singleton.
- **Identity entities in Infrastructure:** `ApplicationUser` and `ApplicationRole` live in `MyMarina.Infrastructure.Identity` (not Domain) because they depend on ASP.NET Core Identity. Domain entities reference users by `UserId` (Guid) only.
- **Primary keys:** UUID v7 (`Guid.CreateVersion7()`) — time-ordered, B-tree friendly, globally unique.
- **Customer accounts:** `CustomerAccount` (not "Customer") — multiple `CustomerAccountMember` records per account with Owner/CoOwner/Member roles.
- **Moorings:** `Slip.DockId` is nullable — null = free-standing mooring/anchorage. `Slip.MarinaId` is always set.
- **Frontend:** React 19 + TypeScript + Vite. Not a .NET project — lives in `src/MyMarina.Web/`, excluded from `.sln`.
- **API types:** OpenAPI spec (auto-generated by ASP.NET Core) → `npm run generate-api` → `src/api/schema.d.ts`. Run after backend changes.
- **Payments:** Manual recording only in MVP. `Payment.PaymentProviderId` and `Payment.PaymentProviderReference` reserved for future provider integration.
- **Customer JWT claims:** JWT claims are derived from the selected `UserContext`. For customers, `customer_account_id` is embedded from the context's `CustomerAccountId`. Portal query handlers filter by both `TenantId` (EF global filter) and `customer_account_id` (explicit WHERE) — never expose cross-customer data. For operators, `marina_id` (nullable) and `tenant_id` determine scope: if `marina_id` is set, operator sees only that marina; if null, operator sees all marinas in tenant.
- **Customer invitation:** `POST /customers/{id}/invite` creates an `ApplicationUser` + a `UserContext` (Customer role, scoped to the tenant + CustomerAccountId) + a `CustomerAccountMember` (Owner role). Returns a generated temporary password. Duplicate email → 409 Conflict.
- **Frontend route architecture:** Two protected layout shells — `operatorRoute` (`id: "operator"`, no path, covers `/`, `/invoices`, etc.) and `portalRoute` (`path: "/portal"`, covers `/portal/slip`, `/portal/invoices`, etc.). Portal child routes use **relative paths** (e.g. `path: "slip"`) so route IDs resolve to `/portal/slip`. Operator child routes use **absolute paths** (e.g. `path: "/invoices"`). Post-login redirect is role-based: `UserRole.Customer (3)` → `/portal`, all others → `/`.
- **EF migration workflow:** When the snapshot is desynced (e.g. after manually deleting migration files), `ef migrations add` generates a partial migration. Always diff the new migration's `Up()` against the model snapshot changes to catch missing `AlterColumn` calls. Write manually if needed.
- This product is NOT LIVE in production yet. It is perfectly acceptable to perform destructive database schema changes.

## Build / Test Commands

```bash
# API — from repo root
dotnet build
dotnet test
dotnet watch --project src/MyMarina.Api

# EF Core migrations (requires Postgres running)
dotnet ef migrations add <Name> --project src/MyMarina.Infrastructure --startup-project src/MyMarina.Api
dotnet ef database update --project src/MyMarina.Infrastructure --startup-project src/MyMarina.Api

# Frontend — from src/MyMarina.Web/
npm install
npm run dev          # Vite dev server on :5173, proxies /api → :5000
npm run build        # Production build
npm run generate-api # Regenerate TypeScript types from running API OpenAPI spec

# Local dev (all services via Docker Compose)
docker-compose up
```

## Architecture Overview

> See `docs/architecture.md` for full details.

```text
src/
  MyMarina.Domain/          # Entities, value objects, enums — no dependencies
  MyMarina.Application/     # Handler interfaces, abstractions (ICommandHandler, IQueryHandler, IMessageBus, etc.)
  MyMarina.Infrastructure/  # EF Core, Postgres, Identity, Hangfire+Redis, multi-tenancy
    Identity/               # ApplicationUser, ApplicationRole (ASP.NET Core Identity)
    Persistence/            # AppDbContext, EF configurations, migrations
    MultiTenancy/           # HttpTenantContext (implements ITenantContext + IMarinaContext + ICustomerContext)
    Messaging/              # HangfireMessageBus (IMessageBus implementation)
    Customers/              # InviteCustomerCommandHandler
    Portal/                 # All portal query/command handlers (GetPortalMe, GetPortalSlip, etc.)
  MyMarina.Api/             # Controller-based API endpoints, middleware, auth, OpenAPI, Scalar UI
    Controllers/            # API controllers (one per feature area); PortalController at /portal/*
    Infrastructure/         # HangfireAuthFilter, etc.
  MyMarina.Web/             # React 19 / Vite SPA (not in .sln)
    src/api/                # client.ts (Axios) + schema.d.ts (OpenAPI codegen output)
tests/
  MyMarina.UnitTests/       # Domain logic, application handler unit tests
  MyMarina.IntegrationTests/ # Full HTTP stack via WebApplicationFactory + Testcontainers (real Postgres)
k8s/                        # Kubernetes manifests (to be added in CI/CD phase)
```

## Tech Stack

- **Backend:** ASP.NET Core (.NET 10), EF Core 10, PostgreSQL, Hangfire + Hangfire.Redis.StackExchange, SignalR, FluentValidation, Scrutor
- **Frontend:** React 19, TypeScript 6, Vite 8, TanStack Router, TanStack Query, shadcn/ui, Tailwind CSS v4, Zustand, Zod, React Hook Form
- **Infra:** Kubernetes, Docker, nginx-ingress, cert-manager, GitHub Actions, ghcr.io
